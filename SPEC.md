# Web2–Web3 Identity Verification System Spec

## Overview
This document defines the on-chain data model, interfaces, and behavioral rules for identity verification results generated by zkVM. For zkVM-specific derivations and output schema, see `ZKVM_SPEC.md`.

## Terminology
- Provider: A Web2 platform where identity attributes originate. Identifier: `providerId`.
- Provider Criteria: A specific verification rule set under a provider. Identifier: `providerCriteriaId`.
- Identity Nullifier: A masked identity identifier referenced on-chain.
- Web3 User: An EVM wallet address.
- Verification Result: The on-chain stored result keyed by `proofUser` and `providerCriteriaId`.

## zkVM Interface Boundary
- The on-chain contract accepts zkVM proof outputs and verifies the proof.
- Exact zkVM field derivations and output encoding are defined in `ZKVM_SPEC.md`.

## On-Chain Storage Model
- Identity binding:
  - Mapping of `(providerId, identityNullifier) -> user`.
- Results:
  - Mapping of `(user, providerCriteriaId) -> [results]` (append-only list).
  - Mapping of `(user, providerCriteriaId) -> latestResultId` (or latest index).
  - Stored result records expose `timestamp` as part of queryable result data.
- Replay protection:
  - Sets for used `timestamp` in replay domain scope.
  - Mapping/set for used nonce replay keys, where nonce checks are domain-scoped (not globally scoped).
- Unbind control:
  - `lastBindTime` per `(identityNullifier)` or `(user, providerId)` as needed.
  - `lastResultSubmissionTime` per `(user, providerCriteriaId)` or `(user, providerId)` as needed.
  - `cooldownWindow` configuration.
  - Optional lock registry or flag indicating downstream lock status for a nullifier or result.

## Interfaces
Function signatures:
- `unbindIdentity(user, identityNullifier)`
- `submitVerificationResult(proof)`
- `reportDownstreamUsage(identityNullifier, inUse)`
- `getLatestResult(user, providerCriteriaId)`
- `getAllResults(user, providerCriteriaId)` (optional)

## Unbind Rules
- `unbindIdentity` is callable only by `proofUser` (the bound user).
- Unbind is blocked if the cooldown window since last bind or last result submission has not elapsed.
- Unbind is blocked if any downstream system has locked the result.
- On successful unbind, all historical data for the nullifier is wiped (binding and associated results) and corresponding downstream lock state is cleared in the same lifecycle.

## Downstream Lock Model
- Verification results are service-blind reusable: multiple downstream services may lock usage of the same result without requiring proof regeneration.
- `reportDownstreamUsage` is accepted only from whitelisted downstream services.
- For lock creation (`inUse = true`), the contract validates that `identityNullifier` is actively bound.
- Locks are tracked by `(service, identityNullifier)` ownership; unlock is allowed only by the same service for that `identityNullifier`.
- Unlock requires owned lock existence and does not require additional active-result context checks.
- Global safety lock applies per `identityNullifier`: unbind is blocked if any active downstream lock exists for that nullifier.
- No per-service user signature is required for lock creation; policy is `whitelisted service + bound identityNullifier`.
- Each service has at most one lock per `identityNullifier`.
- No TTL-based auto-unlock is applied; users must coordinate with locking services for explicit unlock.
- Any privileged wipe or migration path that removes binding/result data must also clear corresponding downstream lock state.
- A governance-controlled emergency lock-resolution path exists for exceptional deadlock cases, with strict authorization and mandatory audit events.

## Whitelist Operations
- Whitelist management supports `addWhitelistedService`, `removeWhitelistedService`, and `setWhitelistedServiceStatus`.
- Service status model includes at least `ACTIVE`, `PAUSED`, and `REVOKED`.
- Only governance-authorized roles may change whitelist membership or status.
- `reportDownstreamUsage` is accepted only from `ACTIVE` whitelisted services.
- Emergency revoke is supported for compromised services with mandatory reason metadata and audit events.
- Revoked services cannot create new locks; handling of their existing locks is governance-resolvable and auditable.

## Relayer Support
- Transactions may be submitted by relayers.
- Storage is always keyed by `proofUser` from the zkVM output, not by transaction sender.

## Binding Behavior
- There is no manual bind operation; binding occurs only via successful on-chain verification result submission.
- On a successful `submitVerificationResult`, if no binding exists, the contract auto-binds `identityNullifier` to `proofUser` (provider-scoped).
- `identityNullifier` is derived from `(providerId, providerUserId)`, so provider-scoped enforcement is correct by construction.
- Multi-criteria proofs are supported; on-chain stores `providerOutput` as opaque bytes while zkVM defines encoding/decoding for downstream interpretation.
- Binding multiplicity is provider-specific and enforced during auto-binding.
- Example policy: `Binance` allows at most one active nullifier per user; providers other than `Binance` (including `GitHub`) allow multiple active nullifiers per user if each nullifier is free.
- A user may hold active bindings for multiple providers concurrently (for example, Binance + GitHub).
- The full `providerOutput` blob is always stored on-chain for downstream business-logic usage.
- `providerOutput` must satisfy configured size limits and versioned/schema-prefixed envelope validation.
- Allowed `providerOutput` schema/version is bound to `providerCriteriaId`.

## Result Selection
- The latest stored result per `(user, providerCriteriaId)` is considered the active result.
- Full-history retrieval must be available via at least one path: on-chain `getAllResults` (if implemented) or event-based indexing.
- If `getAllResults` is omitted, event-based indexing is required for full chronological history retrieval.

## Replay Protection Scope
- The contract defines a canonical replay domain `ReplayDomain`.
- `ReplayDomain` includes at least `chainId`, `contractAddress`, `proofUser`, `identityNullifier`, and `providerCriteriaId`.
- For proxy-based deployments, `contractAddress` in `ReplayDomain` uses the proxy address (`address(this)` at execution).
- Nonce uniqueness is enforced per `ReplayDomain`, not globally.
- Reuse of the same nonce is rejected only when reused within the same `ReplayDomain`.
- Timestamp replay checks are also enforced per `ReplayDomain` (not globally).
- Reuse of the same timestamp is rejected only when replayed within the same `ReplayDomain`.
- `proofHash` is optional telemetry and is not required for replay enforcement.
- This prevents false rejects across concurrent proofs for different users/criteria while preserving replay resistance in the same verification context.

## Privacy and Determinism Guarantees
- No raw Web2 user IDs are stored or emitted on-chain.
- All identity references use `identityNullifier`.
- Determinism guarantees for zkVM outputs and identifier derivations are specified in `ZKVM_SPEC.md`.

## On-Chain Requirements Summary
- Standardized, deterministic error codes are used for all rejection paths.
- Downstream usage is reported via `reportDownstreamUsage`, restricted to whitelisted callers.
- The nullifier is considered locked if any whitelisted service reports `inUse = true`; multiple services may use a result concurrently.
- Locks are nullifier-bound (`identityNullifier`) and owner-bound (`service`).
- User per-service signatures are not required for lock creation; bound-nullifier validation is required.
- No TTL-based auto-unlock is applied; users must coordinate with locking services for explicit unlock.
- Governance includes emergency deadlock-resolution workflow and accountability for exceptional unlock actions.
- Whitelist operations include role-based management, service status enforcement, emergency revoke, and auditable status transitions.
- `providerOutput` is stored as opaque bytes; downstream services interpret according to zkVM-defined encoding.
- Full `providerOutput` blob is stored on-chain (no output-hash substitution).
- `providerOutput` is validated by size limits, envelope integrity, and schema/version allowlist per `providerCriteriaId`.
- Nonce replay protection is domain-scoped (not globally unique) to avoid false rejection across independent users/criteria.
- Performance evaluation includes gas cost per operation, storage growth, and query latency with baselines.
- Governance defines approval and versioning for provider/criteria changes.
- Governance also controls provider-specific binding multiplicity policy.
